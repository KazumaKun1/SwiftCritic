# PRD — SwiftCritic

## Overview

SwiftCritic is a native iOS app that lets Swift developers practice code review skills by reviewing AI-generated code snippets and receiving structured feedback on the quality of their review. The app is offline-first, powered by Apple's on-device Foundation Models framework — no account, no network, no data leaving the device.

**Target user:** Swift developers at any level who want to sharpen their code review instincts — whether preparing for a new job, leveling up on their team, or building habits around clean code principles.

---

## Goals

- Help developers practice recognizing real code issues (logic bugs, naming problems, security flaws, architectural smells)
- Provide immediate, structured feedback so users know not just if they were right, but *why*
- Keep everything offline — no account required, no network dependency

---

## Platform & Requirements

- **Platform:** iOS 26+ (Foundation Models framework is guaranteed at this version)
- **Language scope:** Swift only (v1)
- **AI:** Apple Foundation Models framework (on-device ~3B parameter model)
- **Persistence:** SwiftData (local only)
- **Connectivity:** Offline-first; no online features in v1

## Features

### 1. Challenge Configuration

Before starting a session, the user selects:

- **Category:** Naming / Logic / Security / Memory / Architecture
- **Difficulty:** Easy / Medium / Hard (see difficulty model below)

The session retains these settings. On retry, the same difficulty and category are kept automatically. The user can exit the session at any time to change settings.

#### Difficulty Model

Difficulty covers two axes combined into a single setting:

| Difficulty | Issue Count | Visibility |
|------------|-------------|------------|
| Easy       | 1 issue     | Surface — spotted at a glance |
| Medium     | 2–3 issues  | Subtle — requires careful reading |
| Hard       | 4+ issues   | Hidden — requires reasoning through logic or flow |

This means Hard challenges are not just longer — the issues are intentionally harder to see without thinking them through.

---

### 2. Review Screen (Core Loop)

A code snippet is generated by the on-device model and presented for review. For v1, all challenges are single-file. Multi-file challenges are deferred to v2 (see Technical Notes).

**Feedback input — two layers:**

1. **Inline comments (required):** Users tap or long-press to highlight a line or range, which opens a small comment sheet. They must leave at least one inline comment to submit. This forces specificity and mirrors real PR review habits.
2. **Overall comment (optional):** A general summary field at the top of the review — for observations that span the whole snippet rather than a specific line.

Users can add multiple inline comments across the snippet before submitting.

**Generation (Turn 1 — internal):**
The app prompts the Foundation Models session to generate a code snippet with the appropriate number of intentional issues for the selected difficulty and returns a `@Generable` struct containing:
- The code string
- A labeled list of issues (description, location hint, category, severity, visibility level)

The issue list is stored privately and never shown to the user during review.

---

### 3. Result Screen

Shown after the user submits their review. The app sends the user's comments back to the same Foundation Models session (Turn 2) and asks it to evaluate against the known issue list.

Results are presented in three buckets:

- ✅ **Caught** — issues the user correctly identified, with their comment shown alongside the actual issue
- ❌ **Missed** — intentional issues the user did not flag, revealed with explanation
- ⚠️ **False positives** — things the user flagged that were not actually problems

A short paragraph of constructive feedback is generated summarizing the review quality (e.g., "You caught the security issue quickly but tend to overlook naming conventions.").

After viewing results, the user can tap **Try Again** to get a new challenge at the same difficulty and category, or exit to reconfigure.

---

### 4. Playbook Screen (v2)

A reference section users can consult before or during a review. Contains concise guides on:
- Code review principles and what to look for, in what order
- Common design patterns and anti-patterns
- Review rules (e.g., "check what's breaking before visual review")

**Deferred to v2.** Ship the core loop first.

---

### 5. Scoring & Progress (SwiftData)

A local scoring system to track growth over time. No leaderboard or online sync in v1.

**Tracked per session:**
- Overall score (0–100) from the model's evaluation
- Category breakdown: performance across Naming, Logic, Security, Memory, Architecture
- Streaks (consecutive days with at least one completed review)

**History screen:**
A list of past sessions showing date, difficulty, category, and score. Users can tap into any session to re-read the result and their original comments.

The goal is to surface patterns: "You're strong on Logic but consistently miss Security issues."

**Practice Weak Areas (v2):**
A smart button on the challenge configuration screen — "Practice based on weak areas" — that auto-selects the category the user scores lowest in. Deferred to v2 once enough history data exists to make it meaningful.

## Technical Notes

### Foundation Models — Two-Turn Flow

**Turn 1 (generation):**
```swift
let session = LanguageModelSession()

let challenge = try await session.respond(
    to: """
    Generate a short Swift function with \(issueCount) intentional \(visibility) issues \
    in the \(category) category, appropriate for \(difficulty) difficulty. \
    Return the code and a labeled list of every issue.
    """,
    generating: CodeChallenge.self
)
```

**Turn 2 (evaluation, same session — context retained from Turn 1):**
```swift
let evaluation = try await session.respond(
    to: """
    The user reviewed the code you just generated. Here is their feedback:
    
    Overall comment: \(userOverallComment ?? "none")
    Inline comments: \(userInlineComments)
    
    Based on the intentional issues you already know about, evaluate their review.
    Which issues did they correctly identify? Which did they miss? Did they flag anything that wasn't actually a problem?
    """,
    generating: ReviewEvaluation.self
)
```

### Multi-File Challenges

True multi-file support (tab-based UI, separate file contexts) is deferred to v2. The token constraint is the primary reason: the 4,096-token context window must cover the full session — generated code, issue list, user comments, and evaluation. Two files of ~50 lines each consume ~1,000 tokens for code alone, leaving little room for the rest of the session.

**v1 approach:** Simulate multi-file by presenting multiple labeled sections (e.g., `// Protocol`, `// Implementation`, `// Usage`) as distinct panels within one challenge. This gives the feel of reviewing across contexts without the token risk or UI complexity.

### Key Constraints

- **Context window:** ~4,096 tokens per session. Keep generated snippets concise. Hard difficulty should use more issues, not longer code.
- **Model size:** ~3B parameters on-device. Reliable for syntactic issues and common code smells; prompt engineering is important for consistent structured output at Hard difficulty.
- **Structured output:** Use `@Generable` + `@Guide` macros for both generation and evaluation turns to ensure typed, reliable responses.

## Out of Scope (v1)

- Online leaderboard or social features
- User accounts or cloud sync
- Languages other than Swift
- True multi-file challenges (tab UI)
- The Playbook / reference screen
- "Practice Weak Areas" auto-category selection
- Push notifications or reminders
